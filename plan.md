# Plan for corepost-server

## Роль репозитория

Сервер управления состоянием устройств, регистрацией, выдачей предзагрузочного токена, panic-lock и post-boot политикой.

## Текущее состояние

- Реализация сведена к одному `main.py` без тестовой структуры и без dependency manifest.
- HMAC-аутентификация в коде использует `deviceId` и `emergencyId` как секреты вместо `token`, хотя документация обещает другое.
- Конфигурация хранится в `server.conf`, секреты не разведены по окружению.
- Нет тестов для ключевых сценариев из ПЗ.
- Ветка всё ещё `master`.

## Что сделать

- Пересобрать проект как нормальный Python-пакет: `pyproject.toml`, зависимости, структура `src/`, отдельный модуль конфигурации.
- Исправить модель секретов:
  - клиентский HMAC должен использовать device token;
  - mobile flow должен получить отдельный секрет или чётко описанный derivation flow;
  - install/mobile/preboot/agent должны работать по одному контракту.
- Вынести БД и конфиг в управляемые пути, подготовить миграции схемы.
- Уточнить API для post-boot агента: heartbeat/policy/status/recovery без перегруза мобильных endpoint-ов.
- Убрать хардкод `adminToken`, предусмотреть `.env`/env vars и production-safe defaults.
- Подготовить сервер к локальному production-стенду: logging, healthcheck, README по запуску, пример systemd unit или container runbook.
- Написать тесты на `pytest`:
  - регистрация устройства;
  - штатный decrypt/deny-by-default;
  - неверная подпись;
  - просроченная подпись;
  - двухшаговая блокировка;
  - mobile unlock при `userCanUnlock = true/false`;
  - административное восстановление;
  - повторные серии lock/recover;
  - post-boot политика для агента.
- Добавить линтинг и CI для server API regression tests.

## Критерий готовности

- Сервер проходит автоматические тесты на ключевые сценарии из ПЗ.
- Контракт секретов и endpoint-ов стабилен и пригоден для preboot, install, agent и mobile-клиентов.
